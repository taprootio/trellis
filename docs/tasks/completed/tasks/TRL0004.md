# TRL0004 — Build the MCP server (operations as tools)

## Completed

Shipped an MCP stdio server exposing the seven backlog operations as tools —
`list_tasks`, `get_task`, `next_id`, `create_task`, `move_task`, `validate`,
`regenerate` — each a thin adapter over the TRL0002 core. The operations live in
`src/mcp.mjs` (zero-dependency, transport-agnostic, unit-tested directly); the
`@modelcontextprotocol/sdk` + `zod` wiring and stdio transport live only in the
entry point `scripts/trellis-mcp.mjs` (npm `trellis:mcp`). Tools resolve a repo
root per call (`repoRoot` arg) or from the server default (`--repo`/cwd), so one
server can serve any repo. Results carry the `backlog.json` shape as JSON text +
`structuredContent`; domain errors map to `isError` tool results, not protocol
failures.

Mutating tools (`create_task`, `move_task`, `regenerate`) regenerate and validate
the whole backlog before returning and **roll back** on failure (the stated Risk):
create removes the new file; move restores the source and removes the target — so
the repo is never left invalid or with stale artifacts. `move_task` updates the
front-matter and prepends a `## Completed`/`## Removed` note (SPEC §5.2).

Decision: adopted the official `@modelcontextprotocol/sdk` (+ `zod`) — Trellis's
**first runtime dependency**, with a committed `package-lock.json`. The trade was
the zero-dependency streak for protocol robustness and a clean path to TRL0006
(prompts/resources). The vendored/core code (`src/backlog.mjs`, `src/mcp.mjs`)
stays dependency-free; only the server entry imports the SDK. Smoke tests in
`test/mcp.test.mjs` scaffold a throwaway repo via the init scaffolder and drive
every tool, including the create rollback.

Follow-ups: MCP prompts/resources for the process loops (TRL0006); `effort` label
resolution so `create_task` accepts labels (TRL0015); per-tool `outputSchema` and
the broader integration matrix (TRL0011); `npx trellis` bin + packaging (TRL0010).

## Scope

Expose the backlog operations as MCP **tools**, reusing the TRL0002 core
(`src/backlog.mjs`) for all logic so there is one implementation behind both the
CLI and the server.

**Tools** (each returns the `backlog.json` shape or a slice of it):

| tool | args | returns |
| --- | --- | --- |
| `list_tasks` | `status?`, `milestone?` | the backlog header (`prefix`, `milestones`, `nextId`, `counts`) + `tasks[]`, filtered |
| `get_task` | `id` | the task entry **+ raw Markdown `body` + `file`** path |
| `next_id` | — | `{ nextId }` |
| `create_task` | `title`, `summary`, `milestone`, `priority`, `effort` (number), `depends_on?`, `body?` | assigns the next id, writes `active/<id>.md` (scaffolds Scope/Notes/Risks if `body` omitted), then regenerate+validate; returns the created entry |
| `move_task` | `id`, `to` (`completed`\|`removed`), `reason` (required if `removed`), `note?`, `date?` (ISO; default = server today) | moves `active/<id>.md` → `completed/tasks/` or `removed/`, sets `status` + `completed_on`/`removed_on`(+`removed_reason`), prepends a `## Completed`/`## Removed` note; regenerate+validate |
| `validate` | — | `{ ok, errors, warnings }` — config + item + marker validity (read-only) |
| `regenerate` | — | `{ changed[], nextId, counts }` — rewrites stale artifacts |

**Architecture** — keep the testable logic dependency-free and isolate the SDK at
the entry point (mirrors `src/init.mjs` ↔ `scripts/trellis-init.mjs`):

- `src/mcp.mjs` — zero-dependency, transport-agnostic operations over the core.
  Every function takes an explicit `repoRoot` and owns its domain validation
  (unknown id, "reason required when removing", refuse to mutate an already-invalid
  backlog). Unit-tested directly with `node --test`, no transport.
- `scripts/trellis-mcp.mjs` — the MCP entry point; the SDK lives here only.
  `McpServer` + `StdioServerTransport`, one `registerTool` per op with a `zod`
  input shape (boundary validation + the `tools/list` JSON Schema). Each handler
  calls `src/mcp.mjs` and returns the result as JSON **text content +
  `structuredContent`**; a domain error becomes an `isError` tool result.
  Diagnostics go to **stderr** — stdout is the JSON-RPC channel. npm: `trellis:mcp`.
- **Repo targeting:** a default root resolved at startup (`--repo <path>`, else
  `cwd`); each tool also accepts an optional `repoRoot` to override, so one server
  can serve multiple repos — consistent with the core's explicit-`repoRoot` design.

## Notes

- This is the vendor-neutral core — the open protocol is what makes the formula
  portable across Claude, Cursor, Windsurf, Codex, etc.
- **First runtime dependency.** Trellis was zero-dependency through TRL0003; this
  task adds `@modelcontextprotocol/sdk` + `zod` (with a committed
  `package-lock.json`). The trade was made deliberately for protocol robustness
  and to set up TRL0006 (prompts/resources). The vendored/core code
  (`src/backlog.mjs`, `src/mcp.mjs`) stays dependency-free; only the server entry
  point imports the SDK.
- `effort` is the **canonical number** here; label resolution (SPEC §6.2) is
  TRL0015 and not yet in the core, so a label would fail validation.
- Ships smoke tests (`test/mcp.test.mjs`, as TRL0003 did). Boundaries: MCP
  **prompts/resources** are TRL0006, the **`npx trellis` bin/packaging** is TRL0010,
  and the **full test matrix** is TRL0011.

## Risks

- **Write operations racing the generator.** Every mutating tool (`create_task`,
  `move_task`, `regenerate`) applies its change, then re-reads + validates the
  whole backlog and regenerates every artifact before returning. On any validation
  failure it **rolls back** to the pre-call state, so the repo is never left
  invalid or with stale generated files (mirrors init's no-partial-write ethos).

# TRL0002 — Build the config-driven generator core

## Completed
Extracted the generator into a reusable, zero-dependency core at `src/backlog.mjs`
(`loadConfig`, `parseFrontMatter`, `readBacklog`, `nextId`, `buildBacklogJson`,
`generateArtifacts`) — each takes an explicit `repoRoot`, so the CLI and the
future MCP server (TRL0004) share one implementation. `scripts/backlog-readme.mjs`
is now a thin CLI at the same path, leaving the npm scripts and CI gate unchanged.
The refactor is behavior-preserving: regeneration is byte-identical, so `--check`
stayed green with no artifact churn.

Added config shape validation (idPrefix / idWidth / milestones / priorities /
numeric effort values), a `specVersion` mismatch warning (SPEC §8.3), and a
hardened front-matter parser (CRLF, colons in values, quoted strings,
duplicate-key and malformed-line errors). `depends_on` is required on every item
(no silent default). Active items are validated in full against the live config;
completed/removed items are validated for lifecycle integrity (status, ISO close
date, removed_reason, depends_on referential checks) while their historical enum
values (milestone/priority/effort) are not re-validated (SPEC §5.1 / §8.3).

Decision: kept Trellis **dependency-free** rather than adding a YAML library — the
front-matter is a small fixed subset, and zero dependencies preserve the
frictionless drop-in onboarding that is the point of the tool. This revises the
original "replace with a real YAML parser" scope line.

Follow-ups: full unit/integration tests live in TRL0011; effort-scale object-form
resolution in TRL0015.

## Scope
- Promote `scripts/backlog-readme.mjs` into a packaged module driven entirely by
  `backlog.config.json`.
- Replace the bootstrap front-matter parser with a real YAML parser.
- Expose `validate`, `generate`, and `next-id` as library functions the CLI and
  MCP server both call.

# TRL0025 — Git-derived task history and a materialized history.json

## Scope
- Core deriver + `trellis history [<id>]` CLI: run `git log --follow` over a task
  file by id (resolving the path across `active/`, `completed/tasks/`, `removed/`)
  and return structured entries `{ id, date, author, subject, commit }`,
  newest-first. `--follow` is required so history survives the active→completed move.
- Materialize `history.json` — per repo, keyed by task id — as a derived report a
  static viewer can read without invoking git at serve time. Generate it on demand /
  at site-build (CI has the full clone), not on every edit.
- **Determinism carve-out:** history is volatile by nature (timestamps, authors),
  so it is explicitly **not** part of the deterministic `--check` gate (SPEC §8).
  Add a short spec section for "derived, non-gated reports" so this does not
  contradict the byte-identical-output rule. `backlog:check` must not depend on git
  history.
- MCP (`src/mcp.mjs`): a read-only `history` tool (by id, or whole-repo) returning
  the same structured entries.
- Optional reason enrichment: support a commit trailer (e.g. `Trellis-Reason:`)
  surfaced as a `reason` field; otherwise the commit subject is the reason. The
  PR-title standard (TRL0016) already makes squash-merge subjects clean, so the
  default is useful with no new convention required.

## Notes — import and product direction
- Imported items (TRL0021/TRL0022) get their git history from the import commit
  forward; the deriver must handle single-commit / shallow histories gracefully
  (show the creation/import event; do not assume depth). Pre-Trellis history stays
  in the source system — git, not Trellis, is the deep record, by design.
- Forward shape: a per-repo `history.json` keyed by id is exactly what a later
  cross-repo Trellis API / Taproot static deploy would aggregate. Keep the schema
  stable and repo-scoped — but do **not** build the API or multi-repo wiring here.
- Captures hand-edits for free: because it derives from committed diffs (not from
  tooling hooks), a manual front-matter edit is recorded like any other change — no
  parallel log to drift.

## Risks
- History quality tracks commit/PR hygiene; thin or over-squashed commits yield
  thin overviews. The PR-title lint helps; the optional reason trailer is the escape
  hatch.
- A materialized `history.json`, if committed, can look stale vs git — make clear it
  is a regenerable cache, never authoritative (git is). Prefer build-time generation
  over committing it; if committed, mark it generated and keep it out of `--check`.
- A pure static host serving only files can't derive live — which is exactly why we
  materialize at build time. Ensure the build step, not the runtime, owns derivation.

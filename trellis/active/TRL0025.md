# TRL0025 — Git-derived task history and a materialized history.json

## Scope
- **Core deriver — new `src/history.mjs`.** Keep `git` out of the deterministic
  core (`src/backlog.mjs`): a separate module structurally guarantees
  `backlog:check` never depends on git history. It reuses `paths()` / the id regex
  from the core, resolves an id across `active/`, `completed/tasks/`, `removed/` by
  probing (no full-backlog validation needed — history is a read/forensic tool),
  and runs `git log --follow` over the task file. `--follow` is required so history
  survives the active→completed move (verified: it also tracks the `PL→TRL` prefix
  migration and the `docs/tasks → trellis/` relocation).
- **Entry shape:** `{ id, commit, date, author, subject, reason }`, newest-first.
  `commit` = full SHA (`%H`); `date` = author date as full ISO 8601 (`%aI`);
  `author` = `%an`; `subject` = `%s`. Derived with `\x1f`/`\x1e` field/record
  delimiters so subjects parse unambiguously.
- **`trellis history [<id>]` CLI** — `scripts/trellis-history.mjs` +
  `npm run trellis:history`. Logic stays in `src/`; the script is a thin wrapper
  (codebase pattern). Flags: `--repo <path>` (default cwd, like the MCP server),
  `--json` (structured output to stdout), `--write` / `--out <file>` (materialize
  `history.json`). With an `<id>` → that task's entries; without → whole repo.
- **Materialize `history.json`** — `{ "generated": true, "tasks": { "<id>":
  [entries…] } }`, keyed by task id, every known id present (`[]` if uncommitted).
  Written to the backlog root (`<tasksDir>/history.json`) by `trellis history
  --write`. Generated on demand / at site-build (CI has the full clone), **not** on
  every edit and **not** wired into `backlog:readme` / `backlog:check`. Whole-repo
  derivation is one `git log --follow` per task file (`--follow` allows a single
  pathspec) — fine at this scale, documented.
- **Persistence: build-time, not committed.** Add `trellis/history.json` to
  `.gitignore` so a local run is never committed-and-stale; the static deploy
  regenerates it in CI before publishing. Scaffold the same ignore in `trellis
  init` only if init already manages `.gitignore`.
- **Determinism carve-out (SPEC).** History is volatile (timestamps, authors), so
  it is explicitly **not** a deterministic generated artifact (SPEC §8) and **not**
  part of `--check`. Add SPEC **§8.4 "Derived, non-gated reports"** (plus a pointer
  from the §8 intro) defining the class — regenerable, not authoritative (git is),
  generated on demand / at build time — so it does not contradict the
  byte-identical-output rule. This is an additive, optional capability →
  **bump the spec to 2.2.0** (SPEC header, `SPEC_VERSION`, this repo's
  `backlog.config.json`, AGENTS prose, and the version-tied `init` test).
- **MCP (`src/mcp.mjs`): read-only `history` tool.** `id` optional — by id →
  `{ id, entries }`; whole-repo → `{ tasks: { <id>: [entries…] } }`. Same structured
  entries as the CLI; never writes. Register in `scripts/trellis-mcp.mjs` (TOOLS +
  help). A non-git repo throws a `TrellisError` (mapped from the deriver's typed
  `HistoryError`).
- **Reason enrichment.** `reason` = the `Trellis-Reason:` commit trailer when
  present (`%(trailers:key=Trellis-Reason,valueonly)`), else the commit subject —
  so the field is **always** present and a consumer reads one field. The PR-title
  standard (TRL0016) already makes squash-merge subjects clean, so the default is
  useful with no new convention required.

## Notes — import and product direction
- Imported items (TRL0021/TRL0022) get their git history from the import commit
  forward; the deriver must handle single-commit / shallow histories gracefully
  (an empty history → `[]`, never a crash; squash-merges legitimately yield a
  single entry). Pre-Trellis history stays in the source system — git, not Trellis,
  is the deep record, by design.
- Forward shape: a per-repo `history.json` keyed by id is exactly what a later
  cross-repo Trellis API / Taproot static deploy would aggregate. Keep the schema
  stable and repo-scoped — but do **not** build the API or multi-repo wiring here.
- Captures hand-edits for free: because it derives from committed diffs (not from
  tooling hooks), a manual front-matter edit is recorded like any other change — no
  parallel log to drift.

## Risks
- History quality tracks commit/PR hygiene; thin or over-squashed commits yield
  thin overviews. The PR-title lint helps; the optional reason trailer is the escape
  hatch.
- A materialized `history.json`, if committed, can look stale vs git — make clear it
  is a regenerable cache, never authoritative (git is). We resolve this by **not
  committing it** (gitignored) and generating at build time; the `generated: true`
  marker is belt-and-suspenders if a consumer ever does commit it.
- A pure static host serving only files can't derive live — which is exactly why we
  materialize at build time. The CLI (`--write`), not any serve-time runtime, owns
  derivation.
- Git invocation is the new failure surface: no `git` / not a work tree → a clear
  typed error (CLI exits non-zero; MCP returns an `isError` result), not a stack
  trace. `id` is regex-validated before any path is built and git runs via an
  arg-array (no shell), so a crafted id cannot traverse paths or inject arguments.

# TRL0035 — Dogfood round 2: re-onboard Taproot with the built-in profile after the fixes

## Scope
- Once TRL0031–TRL0034 are shipped in a published patch, re-onboard `../taproot`
  (clean `main`) in a **fresh, unbiased context** using only `npx @taprootio/trellis`
  and the **built-in** `taproot-ai-backlog` profile — no custom mapping.
- Verify the friction from the first dogfood is gone: import is `check`-green with the
  built-in profile; generated indexes aren't imported as tasks; source ids are
  preserved with the new-id floor set (TRL0032); summaries read cleanly (TRL0033); the
  scaffolded `team.json` is inert (TRL0034); and the scaffolded CI gate is green.
- Capture a fresh report. If it's clean, this closes the loop the TRL0010 dogfood
  opened.

## Notes
- The first dogfood (TRL0010) proved the onboarding *mechanism* but needed a
  hand-written mapping and surfaced id / summary / scaffolding friction. This task
  proves the built-in, no-mapping path a real consumer would actually take.
- Run it the same way: a separate context inside taproot, package-only (no reading the
  Trellis source), honest verbatim friction report.

## Risks
- Gate on the fixes being **released** to npm, not just merged — the dogfood installs
  the published `npx` package, so it must run against the patched version (≥ 0.2.0),
  or it will re-hit the already-known issues.

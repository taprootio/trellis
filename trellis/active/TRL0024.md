# TRL0024 — Drop the README "Next task ID"; make backlog.json the canonical source

## Scope
- Generator (`src/backlog.mjs`, `generateArtifacts`): the `readmeBody` becomes just
  the milestone `block` — drop the `## Next task ID` section; the milestone tables
  stay between the markers. Keep `const next` and the `nextId: next` return value
  (callers and `buildBacklogJson` still use it). Leave `nextId` in
  `buildBacklogJson` untouched — `backlog.json` stays canonical.
- Spec: update §3 (ids are assigned from the `nextId` published in the generated
  `backlog.json` (§8.2), not the README) and §8.1 (the README no longer publishes
  the next id; `backlog.json` carries it). Keep the normative wording
  **tool-agnostic** — cite `backlog.json` only, not the reference MCP `next_id`
  tool (SPEC.md is vendored into any repo; the MCP server is the reference impl's
  concern, so that note lives in Risks below).
- Version: bump the spec **minor, 2.2 → 2.3**, in SPEC.md (`Version:` line),
  `SPEC_VERSION` (`src/backlog.mjs`), `trellis/backlog.config.json`, and the
  "targets `specVersion …`" line in AGENTS.md.
- Regenerate this repo (`npm run backlog:readme`) so `trellis/README.md` drops the
  section, and confirm `npm run backlog:check` is green.

## Notes — tests
- `test/backlog.test.mjs` has **no** existing "Next task ID" assertion to update
  (and does not assert `nextId` today), so this *adds* regression guards rather than
  editing one: the generated README no longer contains "Next task ID", and
  `buildBacklogJson(...).nextId` is still emitted.
- The version bump has two test couplings: `test/init.test.mjs` asserts
  `cfg.specVersion === "2.2"` (what `trellis init` writes) → update to `"2.3"`; two
  `test/history.test.mjs` fixtures carry `specVersion: "2.2"` → bump to `"2.3"` for
  honesty (harmless either way — the major matches, so no warning).

## Notes — import
- The importer already sources the next id from the core
  (`nextId(data.ids, cfg)` in `src/import.mjs`), not by parsing the README — so
  removing the README section needs no import change (verified). Add a guard:
  import into a **non-empty** target and assert the assigned ids continue from the
  core `nextId` (today's import tests only cover empty, freshly-init'd targets).
- `trellis init`'s README skeleton (`readmeSkeleton`, `src/init.mjs`) emits only the
  generated markers (no literal "Next task ID"), so no skeleton change is needed
  (verified).

## Risks
- Browsing the repo on a forge loses the at-a-glance next id in the README.
  Mitigated: `backlog.json` carries it and the MCP `next_id` tool returns it — the
  audience already uses both.
- This edits normative spec text (§3/§8.1); keep the wording precise that
  `backlog.json` is now the published source of the next id.

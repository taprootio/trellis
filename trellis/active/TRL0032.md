# TRL0032 — Importer: preserve sound source ids, with a configured new-id floor

## Scope
- Add an import mode that **preserves source ids** (e.g. Taproot `TR00031` → Trellis
  `TR00031`) when the source id parses to the target's `idPrefix` / `idWidth` and the
  item validates — instead of always assigning fresh-sequential ids.
- Introduce a configured **new-id floor**: after a preserving import, the `nextId`
  used for organically-created tasks starts at a configured number *above* the
  imported range, so the preserved band and the new-id band never overlap. The gap
  between them is deliberate space to resolve collisions organically.
- **Collision handling:** when two sources claim the same id, keep the first and
  reassign the collider into the new-id band with a reported remap — never silently
  overwrite.
- Preserving ids also retires the stale-**prose** id-reference problem for same-prefix
  migrations (if ids don't change, body references stay valid). When ids *do* change
  (cross-prefix or a collision), **report** the items whose bodies name a remapped id
  so they can be fixed.

## Notes
- Motivated by the TRL0010 taproot dogfood: fresh-sequential assignment (`TR00031` →
  `TR00001`) broke the repo's identity and left body text pointing at the wrong ids.
- Maintainer's design sketch: "create a space for old ids, and have new ids begin at a
  configured number" — reserve the imported band, floor the new-id counter above it,
  leave a gap for collisions.

## Risks
- Define the "data is good enough" bar precisely: id parses to the target format, no
  duplicate, required fields present. Fall back to fresh-assignment (reported) when it
  isn't met, rather than importing a malformed id.
- `nextId` is currently computed as `max(existing) + 1`; a configured floor changes
  that contract. Keep it backward-compatible and opt-in for repos that don't set one.

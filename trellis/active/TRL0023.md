# TRL0023 — Team roster and task ownership (owner + collaborators)

## Scope
- New authored roster file `trellis/team.json`, an **object** `{ "members": [ … ] }`
  (the wrapper leaves room for future top-level keys — the cross-repo sync
  direction). Kept separate from `backlog.config.json` so the core vocab stays
  stable. It lives at the **fixed** `trellis/` config home (next to
  `backlog.config.json`), independent of `tasksDir`. Each member has a stable
  `handle` (the key used in front-matter; charset `[A-Za-z0-9._-]` so it survives
  inline-list serialization), a `name`, an optional `email`, and a `status` of
  `active` or `inactive` (default `active`). Loaded and validated next to the
  config in `src/backlog.mjs` (`loadRoster`) with actionable errors (unknown keys,
  duplicate handles, bad status, bad handle). **An absent `team.json` is an empty
  roster, not an error** — backward-compatible; only *using* owner/collaborators
  requires roster entries. A malformed roster is a fatal, config-class error
  (fails `--check`).
- Two new optional front-matter fields: `owner` (one member handle) and
  `collaborators` (a list of handles), both resolving against the roster.
- Validation mirrors the milestone/priority pattern (SPEC §5.1, §8.3): on
  **active** items an owner/collaborator must be an **active** member; on
  **completed/removed** items the values are historical and not re-validated, so a
  member who has since gone `inactive` (or left the roster) still validates. Both
  fields are optional everywhere — no backfill of existing tasks required.
- Generator (`src/backlog.mjs`): add an **Owner** column to the active README
  table (`ID | Title | Owner | Priority | Effort`, showing the handle) and `owner`
  (string|null) + `collaborators` ([] if none) to every `backlog.json` task entry,
  for active and closed items alike. Keep the table legible — collaborators ride
  along in `backlog.json` and the item file, not the README table.
- MCP (`src/mcp.mjs`): accept optional `owner`/`collaborators` in `create_task`
  and `move_task` (dedupe collaborators; membership enforced via the existing
  re-read-and-roll-back path, like `depends_on`); surface them in `get_task` /
  `list_tasks` (automatic via `backlog.json`).
- `trellis init` (`src/init.mjs`): scaffold a `team.json` stub (one example active
  member) at the fixed config home and mention the roster + owner/collaborators in
  the AGENTS block — idempotent, no clobbering, `--force`-overwritable, reported
  under created/skipped (authored, not generated).
- Spec: add a roster section (§7.2, the authored `team.json` shape), extend the
  §5.1 schema table with `owner`/`collaborators`, document them in the §8.2
  `backlog.json` shape, and record the active-vs-historical rule in §8.3. Bump
  `specVersion` to **2.1** — additive (SPEC §9).
- Dogfood: add `trellis/team.json` with one active member (`jereme` / Jereme Evans
  / jereme.evans@gmail.com) and set `owner: jereme` on this task; other active
  tasks stay unassigned.

## Notes — import (coordinate with TRL0022, landed)
- The importer (`src/import.mjs`) maps source assignees too: add `fields.owner` /
  `fields.collaborators` extractors and an identity **`remap.owner`**
  (case-insensitive) that applies to both `owner` and each `collaborator` (one
  identity space), parallel to `remap.priority` / `remap.milestone`. Reuse the
  `resolveOrCarry` shape so active items must resolve to an active roster member
  while closed items carry the historical value with a warning.
- Unmapped-owner policy (decided): for an **active** item, resolve via
  `remap.owner` / a direct case-insensitive roster match, then fall back to
  `defaults.owner` (active-items-only), else **drop to unassigned with a warning**
  — never silently invent a member. An unresolved **collaborator** on an active
  item is likewise dropped with a warning. On **closed** items, an unresolved
  owner/collaborator is carried as a historical value with a warning. Carrying
  owners therefore requires the target repo to have a `team.json` roster; roster
  curation stays a manual post-import step.
- Update `docs/import.md` (the full mapping reference) and the built-in profiles
  shipped in TRL0022 (incl. Taproot's `ai-backlog`) so imported backlogs can carry
  owners and still land `--check`-green.
- This task depended on TRL0022: both extend the mapping/profile surface. The
  roster/schema/generator work itself is import-independent.

## Risks
- Roster location: a separate `team.json` at the fixed config home is the
  deliberate call (richer, faster-changing data; stable core config; a clean unit
  to sync across repos in the future product direction) — but it adds an authored
  file the tooling must find and validate. Its loader and error messages are kept
  as crisp as the config's.
- Over-validation would hurt: making `owner` required, or re-validating historical
  owners, forces backfills and breaks the archive. Both fields stay optional and
  historical-on-close.
- Identity model: `handle` is the stable key; `name`/`email` are display only.
  Not coupled to any external identity provider — that belongs to the later
  cross-repo Trellis API direction.

# TRL0023 — Team roster and task ownership (owner + collaborators)

## Scope
- New authored roster file `trellis/team.json` (kept separate from
  `backlog.config.json` so the core vocab stays stable): a list of members, each
  with a stable `handle` (the key used in front-matter), a `name`, an optional
  `email`, and a `status` of `active` or `inactive`. Loaded and validated next to
  the config in `src/backlog.mjs` with actionable errors (unknown keys, duplicate
  handles, bad status).
- Two new optional front-matter fields: `owner` (one member handle) and
  `collaborators` (a list of handles), both resolving against the roster.
- Validation mirrors the milestone/priority pattern (SPEC §5.1, §8.3): on
  **active** items an owner/collaborator must be an **active** member; on
  **completed/removed** items the values are historical and not re-validated, so a
  member who has since gone `inactive` (or left the roster) still validates. Both
  fields are optional everywhere — no backfill of existing tasks required.
- Generator (`src/backlog.mjs`): add an **Owner** column to the active README
  table and `owner` + `collaborators` to every `backlog.json` task entry. Keep the
  table legible — collaborators ride along in `backlog.json` and the item file, not
  the README table.
- MCP (`src/mcp.mjs`): accept `owner`/`collaborators` in `create_task` and
  `move_task`; surface them in `get_task` / `list_tasks`.
- `trellis init` (`src/init.mjs`): scaffold a `team.json` stub (one example member)
  and mention it in the AGENTS block — idempotent, no clobbering, consistent with
  the existing scaffold actions.
- Spec: add a roster section (the authored `team.json` shape), extend the §5.1
  schema table with `owner`/`collaborators`, and record the active-vs-historical
  rule in §8.3. Bump `specVersion` minor — additive (SPEC §9).

## Notes — import (coordinate with TRL0022, in flight)
- The importer (`src/import.mjs`) must map source assignees too: add
  `fields.owner` / `fields.collaborators` extractors and an identity
  **`remap.owner`** (case-insensitive), exactly parallel to `remap.priority` /
  `remap.milestone`. Reuse `resolveOrCarry` so active items must resolve to an
  active roster member while closed items carry the historical value with a warning.
- Decide and document the unmapped-owner policy: an imported active item whose
  owner isn't in the roster should map via `remap.owner`, fall back to
  `defaults.owner`, or drop to unassigned — never silently invent a member. Roster
  curation can stay a manual post-import step.
- Update `docs/import.md` (the full mapping reference) and the built-in profiles
  shipped in TRL0022 (incl. Taproot's `ai-backlog`) so imported backlogs carry
  owners and still land `--check`-green.
- This is why the task depends on TRL0022: both extend the mapping/profile surface,
  so land this after the import-profile work settles to avoid schema churn. The
  roster/schema/generator work itself is import-independent.

## Risks
- Roster location: a separate `team.json` is the deliberate call (richer,
  faster-changing data; stable core config; a clean unit to sync across repos in
  the future product direction) — but it adds an authored file the tooling must find
  and validate. Keep its loader and error messages as crisp as the config's.
- Over-validation would hurt: making `owner` required, or re-validating historical
  owners, forces backfills and breaks the archive. Keep both fields optional and
  historical-on-close.
- Identity model: `handle` is the stable key; `name`/`email` are display only.
  Don't couple to any external identity provider now — that belongs to the later
  cross-repo Trellis API direction.

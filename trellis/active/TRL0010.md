# TRL0010 — Packaging, versioned spec, and docs

## Scope
- Publish an installable package to **public npm** as `@taprootio/trellis`:
  `npx @taprootio/trellis init|import|generate|check|history|mcp` plus the MCP
  server entrypoint. Rename off the earlier unscoped `ai-trellis` placeholder
  everywhere it appears — `package.json` (`name`; `bin` → `trellis`), the CLI/MCP
  `HELP` text in `scripts/*.mjs`, `README.md`, `docs/import.md`, the scaffolding
  strings in `src/init.mjs` (the AGENTS "Loop contract" block and the CI workflow
  it writes), the vendored `templates/trellis/**`, the playbooks, and the tests
  that pin the old strings.
- Publish from **GitHub CI** (`.github/workflows/release.yml`): a push to `main`
  whose `package.json` version isn't yet on npm triggers the publish — the version
  bump, reviewed in its PR, is the release act (no separate tag step; the job
  auto-creates the `v<version>` tag + GitHub Release, and routes prereleases to the
  `next` dist-tag). Auth is a short-lived npm token (`secrets.NPM_TOKEN`) now,
  migrating to npm **trusted publishing** (OIDC) — the job is already OIDC-ready
  (`id-token: write` + `npm publish --provenance`), so the migration is
  configure-the-trusted-publisher-then-drop-the-secret.
- Dogfood the packaged onboarding: onboard a real repo (`../taproot`, on a clean
  main) with the published `npx @taprootio/trellis` (init → import → generate →
  check), with the scaffolded CI gate actually green — proving the consumer
  onboarding path, not just the format (TRL0009 proved format/process on dev
  tooling; this proves the package).

## Notes
- `ai-trellis` was never published (TRL0030 picked that unscoped name only because
  npm's `trellis` was taken); the `@taprootio` scope sidesteps the collision, so
  the placeholder is renamed away wholesale rather than aliased.
- Already done, so not re-litigated here: `SPEC.md` is versioned (`2.3.0`, stable)
  with a compatibility note (§9), and the getting-started + reference docs exist
  (`README.md`, `SPEC.md`, `docs/import.md`). The remaining work is the package
  name, the release pipeline, and the dogfood.
- Stays `active` until the package is actually published and the taproot dogfood is
  green — both are post-merge, collaborative steps.
- Ships an MIT `LICENSE` (copyright "Taproot") — confirm the license + holder
  before the first release.

## Risks
- Semver discipline on the spec vs. the tooling. Version them independently.
- Registry/scope wiring: `@taprootio` must resolve to public npm for both publish
  and `npx` install. CI pins the registry via `setup-node`; this machine's
  `~/.npmrc` currently scope-maps `@taprootio` to GitHub Packages, so it must be
  repointed (or removed) before the local `npx` dogfood resolves.
- Half-rename hazard: the name ships inside scaffolded CI + vendored templates, so
  a missed occurrence would bake the wrong package name into every onboarded repo.

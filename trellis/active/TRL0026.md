# TRL0026 — Importer: resolve legacy close-dates and effort faithfully

## Scope
- Add a **git-commit-date fallback** for `completed_on` (and `removed_on`) when a
  source item has no parseable date header, recording provenance so a derived date
  is distinguishable from an authored one.
- Add a **`Size:`→`effort`** path so legacy items that record size under another
  label resolve to a real effort value instead of the flat default.
- **Flag estimates explicitly:** where no effort signal exists, fill the configured
  default but mark it as estimated, and report counts in the import summary
  (e.g. "N closed items git-dated, M effort-estimated") so nothing silently passes
  as authored.
- Import-time only — the deterministic generator and `--check` stay git-free
  (SPEC §8.4); the one-time importer may read git.
- Update `profiles/taproot-ai-backlog.json` and `docs/import.md`; add a header-less,
  size-only fixture to the import tests.

## Notes
- Proven against Taproot: a dry run refuses because 14 completed items carry no
  `Completed:`/`Created:` header, and several inline-legacy items use `**Size:** N`
  rather than `**Effort:**`. This task is the fix that friction points to.
- Closed-item effort/priority are historical and non-gated, so an estimate is
  cosmetic-but-honest; the flag is what keeps it from being mistaken for an
  authored value.
- A higher-quality, per-item effort estimate (an agent reading each no-signal item)
  lives in the onboarding recipe (TRL0009), not in the dependency-free CLI.

## Risks
- Git reads at import time must degrade gracefully — shallow clones, an uncommitted
  source tree, or single-commit history fall back to the configured default plus the
  estimate flag, and never fail the import.

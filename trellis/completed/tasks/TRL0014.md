# TRL0014 — Bake branch protection + required backlog check into the recipe

## Completed
Baked branch protection into the Trellis recipe. Added `docs/branch-protection.md`
— a forge-agnostic setup recipe: a copy-paste GitHub `gh api` snippet plus the
Settings → Branches UI steps, and concise GitLab / Bitbucket / Azure DevOps
equivalents (the setting that maps to a protected branch + a required check), all
keyed to the pinned `backlog` context, with a closing "verify the gate" step.

`trellis init` now copies the recipe into onboarded repos (via `COPY_FILES`, like
the playbooks) and its post-scaffold output points at it. The emitted workflow and
this repo's `.github/workflows/backlog.yml` carry an explicit `name: backlog`, so
the required-check context is a stable, pinned name rather than an implicit job id.
`SPEC.md` §10 gained a sentence requiring that pinned context, and the AGENTS block
(both this repo's and the init-emitted one) points at the recipe. A new
`test/init.test.mjs` case asserts the doc scaffolds, the workflow is pinned, and
the copied recipe stays portable (no repo-root SPEC link, no author branch prefix).

Decisions (refinement): the helper is documentation + a paste-able `gh api`
command, not a committed executable; the context is pinned via an explicit
`name: backlog` that matches the existing job id, so a repo already requiring the
`backlog` check needs no settings change. Scope item 1 (the §10 mandate) and the
job's pinned context already existed from TRL0001 / TRL0003; this task added the
setup recipe, the non-GitHub coverage, the explicit name pin, and the §10
stable-context sentence.

Follow-up: the scaffolded CI runs `npx trellis check`, red until TRL0010 publishes
the package — a pre-existing, accepted trade-off, not introduced here.

## Scope
- **Setup recipe** — add `docs/branch-protection.md`: a forge-agnostic recipe for
  enabling the gate. GitHub gets a copy-paste `gh api` snippet **and** the
  Settings → Branches UI steps; GitLab, Bitbucket, and Azure DevOps get a concise
  equivalent (the setting that maps to "protected branch" + "required check") so
  the recipe is not GitHub-only.
- **Emit it from init** — `trellis init` (TRL0003) copies `docs/branch-protection.md`
  into the onboarded repo (the same way it copies the playbooks) and points at it
  from its post-scaffold output, so enablement travels with the scaffold. No
  committed executable — the `gh api` command lives in the doc to paste.
- **Pin the required-check context** — give the workflow job an explicit
  `name: backlog` (in init's emitted workflow and this repo's) so the required
  status check is a stable, named context. Document that the context is `backlog`
  — the job name — **not** the workflow display name ("Backlog Hygiene").
- **Spec** — `SPEC.md` §10 already mandates a protected default branch and the
  generator's `--check` as a required status check. Tighten it with one sentence:
  the required check MUST be identified by a stable, pinned context name so a
  workflow/job rename cannot silently drop the gate.

## Notes
- Branch protection is a host repo setting, not committed code — that is why this
  is its own task rather than part of the file scaffolding in TRL0003. TRL0003
  deliberately emitted only the workflow with a pinned-context job and deferred
  enablement here.
- The trellis repo itself is already protected as an ops step, ahead of this task
  generalizing it; this task ships the recipe + the pin, not a change to this
  repo's host settings.
- The onboarded repo's check is `npx trellis check` (red until TRL0010 publishes
  the package); this repo's is `npm run backlog:check`. The recipe names the
  *check context* (`backlog`), which is independent of either command.

## Risks
- Required-check context name drift (workflow/job rename) silently disables the
  gate. Mitigated by the explicit `name: backlog` pin plus documenting the exact
  context string in the recipe and the spec.

# TRL0003 — Build the `trellis init` scaffolder

## Completed
Shipped `trellis init` as a zero-dependency scaffolder: `src/init.mjs`
(reusable `planScaffold` / `applyScaffold`, explicit `targetRoot`, reusing the
TRL0002 core to fill the generated artifacts) plus a `scripts/trellis-init.mjs`
CLI and a `trellis:init` npm script. It onboards any repo to the Trellis layout —
`backlog.config.json`, the `docs/tasks/` tree, the generated
README/indexes/`backlog.json`, `.github/workflows/backlog.yml` (pinned `backlog`
job), the process playbooks + PR template, and an appended AGENTS.md block —
driven by flags (`--prefix` / `--id-width` / `--milestones` / `--priorities` /
`--effort`) with defaults and a TTY-prompt fallback.

Idempotent by contract: it never clobbers (only missing files are written),
`--force` overwrites, `--dry-run` previews, and the AGENTS block is appended only
when its markers are absent. A `node --test` smoke suite (`test/init.test.mjs`)
proves a fresh scaffold validates clean through the core and that re-runs are
no-ops.

Decisions (agreed in refinement): init scaffolds files only and does **not**
vendor the generator — the onboarded repo runs Trellis via the package (TRL0010),
so the scaffolded CI calls `npx trellis check` and is red until that ships, by
design. Branch-protection enablement stays in TRL0014; init only emits the
workflow with a pinned check context. The scaffolded playbooks are copied
verbatim and still mention `npm run backlog:*`; aligning that command surface
with `npx trellis` is a follow-up for TRL0009 / TRL0010.

Follow-ups: TRL0009 dogfoods init on a second repo; TRL0010 publishes the package
and finalizes the CLI surface; TRL0011 owns the full test suite.

## Scope
- Build `trellis init` as `src/init.mjs` (reusable, explicit `targetRoot`, reuses
  the TRL0002 core) plus a `scripts/trellis-init.mjs` CLI and a `trellis:init`
  npm script. Invoked `node scripts/trellis-init.mjs <target> [flags]` (default
  target `.`).
- Scaffold the convention files into the target: `backlog.config.json`; the
  `docs/tasks/{active,completed/tasks,removed}/` layout; the generated
  `README.md`, `completed/index.md`, `removed/index.md`, and `backlog.json`
  (marker skeletons filled by running the core once, so the repo is `--check`-green
  immediately); `.github/workflows/backlog.yml` with a **pinned job name**; an
  **AGENTS.md** backlog section appended as a marked block; and the process files
  `docs/playbooks/{work-task,code-review,pr-draft}.md` and
  `.github/pull_request_template.md`.
- Configure per-repo vocabulary via flags with sensible defaults (`--prefix`,
  `--id-width`, `--milestones`, `--priorities`, `--effort`), falling back to
  built-in `readline` prompts only on a TTY when flags are omitted. Zero new
  dependencies.
- Be idempotent: detect an existing backlog and **never clobber** — create only
  missing files, append the AGENTS block only if its markers are absent, and
  print a per-file `created / skipped / appended` summary. Support `--force`
  (overwrite) and `--dry-run` (report only).

## Out of scope
- **Vendoring the generator.** The onboarded repo obtains the Trellis tool via the
  package (TRL0010); init does not copy `src/backlog.mjs`. The scaffolded CI
  references the package CLI (provisional `npx trellis check`) and is therefore
  red until TRL0010 ships — an accepted, documented trade-off.
- **Branch-protection enablement** (owned by TRL0014). Init only emits the
  workflow with a pinned check context so TRL0014 can require it without drift.

## Notes
- This is the "onboard a repo to the Trellis way" deliverable from the original
  ask. It scaffolds both the backlog mechanics and the process playbooks, so a
  fresh repo gets the full formula in one shot.
- Downstream: TRL0009 dogfoods init on a second repo; TRL0010 turns the `npx`
  story into a published package and finalizes the CLI surface; TRL0011 owns the
  full test suite (this task ships only a smoke test).

## Risks
- Writing into repos with conflicting AGENTS.md content. Append a marked block,
  never rewrite; skip if the markers already exist.
- Clobbering an existing backlog. No-clobber by default; only `--force`
  overwrites; `--dry-run` previews.
- Scaffolded CI references a not-yet-published package (TRL0010), so onboarded
  repos have a failing check until then — an accepted, documented trade-off of
  the files-only approach.

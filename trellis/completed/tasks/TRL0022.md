# TRL0022 — Expose import over MCP and ship reusable source-mapping profiles

## Completed
Generalized the TRL0021 import engine into reusable profiles and wired it into the
two onboarding surfaces, with the engine (`src/import.mjs`) unchanged in logic —
profiles are just mappings, and every new surface is a thin adapter over
`planImport` / `applyImport`.

Shipped a zero-dependency profile registry (`src/profiles.mjs`:
`listProfiles` / `loadProfile` / `loadMappingFile`) and two built-in profiles under
`profiles/`: `taproot-ai-backlog` (the proven reference — bold-inline active items,
header-style closed items, P1–P3, mixed maturity/area milestones; promoted from the
TRL0021 fixture mapping, which the import tests now load so it doubles as a live
regression fixture) and `yaml-frontmatter` (a differently-shaped second source —
full YAML front-matter, no remap — with its own fixture, proving the schema is not
over-fit to Taproot, per the Risk).

Exposed import over MCP as the `import` tool (`importOp` in `src/mcp.mjs`, registered
in `scripts/trellis-mcp.mjs`): `source` plus exactly one of `profile` / `mapping`
(inline), **dry-run unless `apply: true`**, surfacing a refused import as a
`TrellisError` and inheriting the engine's regenerate-then-roll-back contract. Added
the `trellis init --import <path> [--profile | --mapping]` on-ramp (scaffold then
import in one command, apply-ness following `--dry-run`), `trellis import --profile`
/ `--list-profiles`, and aligned relative-`source` resolution to the target repo
across all three entry points. Documented the mapping/profile schema and a
getting-started guide in `docs/import.md`, with two caveats made explicit: remap
targets are the *target* repo's vocabulary, and the engine routes items by source
directory, not a per-item status field.

Decisions (agreed in refinement): a Markdown schema doc, no formal JSON Schema; the
MCP tool dry-run by default (matching the CLI's safe default for a bulk op); the
`init --import` flag over a documented two-step; and the YAML-front-matter shape as
the anti-over-fit second profile. Follow-ups: TRL0009 dogfoods the real Taproot
import using the `taproot-ai-backlog` profile; the profiles double as TRL0011
fixtures.

## Scope
- Expose import over MCP as an `import` tool: an `importOp(repoRoot, args)` in
  `src/mcp.mjs` (added to `OPS`) plus a `zod` registration in
  `scripts/trellis-mcp.mjs`, sharing the TRL0021 engine. Args: `source` (required),
  exactly one of `profile` (named) or `mapping` (inline object), and `apply`
  (**dry-run by default** — `apply: true` writes). It calls `applyImport`, so it
  inherits the regenerate-then-roll-back-on-failure contract the mutating tools
  already follow (TRL0004); `summary.errors` surface as an `isError` result.
- Ship reusable **source-mapping profiles** — mapping objects of the engine's
  existing shape — under `profiles/`, resolved by name through a new
  zero-dependency `src/profiles.mjs` (`listProfiles` / `loadProfile` /
  `loadMappingFile`) shared by both CLIs and the MCP tool:
  - `taproot-ai-backlog` — the proven reference profile (Taproot's
    `planning/ai-backlog/`, the shape TRL0009 imports); promoted from the TRL0021
    fixture mapping, which the import tests then load so the profile doubles as a
    live regression fixture (TRL0011).
  - `yaml-frontmatter` — a generic, differently-shaped example (full YAML
    front-matter, no remap) with its own fixture, to prove the schema is not
    over-fit to Taproot (the Risk).
- Document the mapping/profile **schema** and a getting-started "onboard a repo
  that already has a backlog" guide in `docs/import.md` (Markdown only, no formal
  JSON Schema): every extractor, the built-in profiles as worked examples, and two
  caveats — remap targets must be the *target* repo's configured vocabulary, and
  the engine routes items by source **directory**, not a per-item status field.
- Wire import into onboarding: `trellis init --import <path> [--profile <name> |
  --mapping <file>]` (CLI-only — there is no init MCP tool), composing
  scaffold-then-import in `scripts/trellis-init.mjs`; its apply-ness follows init's
  `--dry-run`. `trellis import` also gains `--profile` and `--list-profiles`.
  Relative `source` paths resolve against the target repo across all three entry
  points (absolute paths unchanged).

## Notes
- The `trellis import` CLI from TRL0021 already unblocks the TRL0009 dogfood; this
  task generalizes the importer — named profiles plus the MCP surface — so any
  MCP-aware client can run an import and disparate repos get a describable on-ramp.
- The engine (`src/import.mjs`) is unchanged in logic: profiles are just mappings,
  and the new surfaces are thin adapters over `planImport`/`applyImport`. The
  profiles double as regression fixtures for the TRL0011 test matrix.
- Built-in profiles assume the **default** Trellis vocabulary as remap targets
  (`Alpha → Beta → v1 → Future`, `High/Medium/Low`, Fibonacci effort); a target on
  a different vocabulary adjusts the profile's `remap`. Documented in `docs/import.md`.

## Risks
- Over-fitting the profile schema to Taproot. Validate it against the second,
  differently-shaped `yaml-frontmatter` source (full YAML front-matter, no remap)
  before calling the schema stable; both profiles import `--check`-green in tests.

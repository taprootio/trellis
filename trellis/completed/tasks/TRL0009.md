# TRL0009 — Dogfood: onboard an existing repo end-to-end

## Completed

Dogfooded Trellis in the `taproot-controls` repository on
`codex/trl0009-dogfood-trellis` and committed the migration as `8641679` (`Adopt
Trellis backlog`). The run initialized an `ESP` backlog, imported 99 legacy
`planning/ai-backlog/` items as `ESP0001` through `ESP0099`, created and closed
`ESP0100 — Adopt Trellis`, reconciled repo guidance, retired the old source tree
with the Trellis retirement path, and passed the local Trellis check plus
Espalier's lint and Vitest gates.

The pass surfaced one recipe-order correction: create the adoption tracker only
after importing an existing backlog. Because imported ids start at the target's
current `nextId`, creating `ESP0100` before import would consume the first id and
shift the imported backlog to `ESP0101+`. This closeout updates the import guide
and `trellis init` next-step text to say "import first, then create the adoption
tracker." It also confirmed the packaging risk: the onboarded CI gate must wait
for TRL0010's package publish before it can run automatically.

## Scope
- Onboard Taproot (Espalier) end-to-end with `trellis init` + `trellis import`:
  prefix `ESP`, backlog root at the default `trellis/` (Taproot's Eleventy build
  publishes `docs/` for the Espalier docs, so the backlog stays out of it).
- Author the canonical **onboarding recipe** as this task's deliverable, and run it:
  1. Branch from `main`.
  2. `trellis init` — the new backlog now exists.
  3. `trellis import` Taproot's `planning/ai-backlog/` via the `taproot-ai-backlog`
     profile; review the id map and per-field warnings, then re-run with `--apply`.
  4. Create the first new item, `ESP0100 — Adopt Trellis`, as the tracker for the
     onboarding itself. For an existing-backlog adoption, this happens after import
     so the imported tasks keep the first available id range.
  5. Reconcile guidance: rewrite the stale "AI Backlog" section in `AI_GUIDELINES.md`
     to point at Trellis (`init` reports it; the agent edits — TRL0027).
  6. Commit the reviewed scaffold/import/reconciliation while the old source tree is
     still intact, then retire the old `planning/ai-backlog/` tree
     (history-preserving, opt-in — TRL0027) in the follow-up closeout.
  7. Work `ESP0100` through the work-task → review loop to prove the loop travels;
     capture any residual friction back into the spec/CLI.

## Notes
- Stress test: Taproot's three duplicate `021`s and mixed active/closed field shapes
  exercise the importer's dedupe, dependency rewrite, and historical-field handling.
  A trial run renumbers cleanly to `ESP0001..ESP0099` with zero dependency errors.
- Friction split out as blocking fixes: legacy close-date + effort resolution
  (TRL0026), init retire/reconcile (TRL0027), scaffold-doc placement under the
  Trellis home rather than `docs/` (TRL0028), and stripping the source id prefix from
  imported titles (TRL0029). A first dogfood pass on dev tooling surfaced 28/29 and
  was reverted; the clean re-run lands after they do.
- The packaged onboarding (real `npx trellis` + a green CI gate) is proven separately
  in TRL0010; this task proves the format/process travels on the dev tooling.
- ESP config decided: prefix `ESP`, milestones `Alpha → Beta → v1 → Future`,
  closed items file under `v1`, closed-item priority defaults to Low, and the 35
  no-signal completed items get a per-item effort SWAG during onboarding.

## Risks
- CI-gating depends on packaging (`npx trellis check`), which ships in TRL0010 (v1).
  Prove the loop locally now and enable branch protection on the `backlog` check only
  after TRL0010 — do not block this task on it.
- Renumbering is intentionally lossy of the source numbers — the fresh-sequential id
  assignment is exactly what dedupes the `021`s. Accepted.
